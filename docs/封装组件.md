# ModelViewer 组件封装设计稿

## 背景

当前 `demo/pages/Demo2.tsx` 已经具备一套完整的“模型展示 + 相机脚本控制”能力，但这些能力目前分散在 demo 页面状态、demo hooks 和库内部组件之间，还没有形成一个适合第三方项目通过 npm 安装后直接 import 使用的高层组件。

本设计稿的目标是：

- 封装一个可对外导出的独立组件
- 第三方用户安装包后可以直接 `import` 组件使用
- 用户传入模型参数和相机脚本参数后即可直接展示模型
- `Demo2` 改造成该组件的使用示例页，而不是继续承担实现职责

目标用法如下：

```tsx
import { ModelViewer } from 'threejs-viewer';

export function Example() {
  return (
    <ModelViewer
      model={{ type: 'url', url: 'https://example.com/model.glb' }}
      cameraScript={{ mode: 'none' }}
      style={{ width: '100%', height: 480 }}
    />
  );
}
```

## 当前 Demo2 能力梳理

### 1. 模型输入

`Demo2` 当前支持 3 类模型输入方式：

- URL 字符串
- 本地单文件 `File`
  - 支持 `.glb`
  - 支持 `.gltf`
  - 如果是 `.gltf`，可额外选择纹理或资源文件
- 本地文件夹 `File[]`
  - 自动从文件夹中寻找 `.gltf` 或 `.glb`
  - 其余文件作为纹理或 buffer 资源参与解析

### 2. 相机脚本模式

`Demo2` 当前支持 3 种模式：

- `none`
- `shot`
- `preset`

### 3. 模式对应数据

- `shot` 模式
  - 传入 `cameraShotJson`
  - 还可附带 `loop`
  - 还可附带 `autoPlay`
- `preset` 模式
  - 传入 `cameraViewPresetJson`

需要注意的是，底层 `CameraScriptController` 实际已经支持两种输入形式：

- JSON 字符串
- 已解析对象

也就是说，对外组件设计不应只支持字符串输入。

## 封装目标

新增一个高层组件：

- 组件名：`ModelViewer`

定位如下：

- `ThreeViewer` 是底层通用 Viewer 组件
- `ModelViewer` 是面向业务和第三方用户的开箱即用组件

也就是：

- `ThreeViewer` 继续保留，适合高级用法
- `ModelViewer` 面向“传参即展示”的主场景

## 对外 API 设计

### 组件导入方式

第一版建议直接从包根入口导出：

```tsx
import { ModelViewer } from 'threejs-viewer';
```

这意味着第三方用户只需要：

1. `npm install threejs-viewer`
2. 在 React 项目中 import 使用

第一版不强制引入子路径导出，例如：

```tsx
import { ModelViewer } from 'threejs-viewer/model-viewer';
```

该方式可作为后续增强项，不作为本期必须目标。

### Props 设计

```ts
type ModelViewerModel =
  | { type: 'url'; url: string }
  | { type: 'file'; file: File; resources?: File[] }
  | { type: 'folder'; files: File[] };

type ModelViewerCameraScript =
  | { mode?: 'none' }
  | {
      mode: 'shot';
      data: string | CameraShot;
      loop?: boolean;
      autoPlay?: boolean;
    }
  | {
      mode: 'preset';
      data: string | CameraViewPreset;
      applyWhen?: 'immediate' | 'afterModelLoaded';
    };

interface ModelViewerProps {
  model?: ModelViewerModel | null;
  cameraScript?: ModelViewerCameraScript;

  grid?: GridConfig;
  pivotPoint?: { x: number; y: number; z: number };
  zoomLimits?: { min?: number; max?: number };
  backgroundColor?: number | string;

  className?: string;
  style?: React.CSSProperties;

  onLoad?: (result: ModelLoadResult) => void;
  onLoadingChange?: (loading: boolean) => void;
  onError?: (
    error: Error,
    context: { stage: 'model-source' | 'model-load' | 'camera-script' }
  ) => void;
}
```

## 为什么采用 `model + cameraScript` 结构

不建议把对外 API 设计成下面这种平铺方式：

```ts
interface BadIdeaProps {
  modelUrl?: string;
  modelFile?: File;
  folderFiles?: File[];
  resourceFiles?: File[];
  mode?: 'none' | 'shot' | 'preset';
  cameraShotJson?: string;
  cameraViewPresetJson?: string;
}
```

原因如下：

- 平铺 props 容易形成互斥组合
  - `modelUrl` 和 `modelFile` 同时传时如何处理
  - `cameraShotJson` 和 `cameraViewPresetJson` 同时传时如何处理
- 第三方用户不容易理解哪些参数是一组
- 内部校验会变复杂
- 文档说明成本更高

采用 `model + cameraScript` 两个主参数的好处：

- 语义清晰
- 互斥关系天然由联合类型表达
- 类型提示更友好
- 对第三方用户更稳定

## 内部实现设计

建议新增以下实现：

- `src/components/ModelViewer.tsx`
- `src/hooks/useResolvedModel.ts`

### `ModelViewer` 职责

- 接收对外 props
- 将 `model` 统一解析成真实可加载的 `modelUrl`
- 管理模型加载结果 `loadResult`
- 自动计算 `modelCenter` 和 `modelRadius`
- 内部组合：
  - `ThreeViewer`
  - `CameraScriptController`
- 统一向外抛出：
  - `onLoad`
  - `onLoadingChange`
  - `onError`

### `useResolvedModel` 职责

- 处理 `model.type === 'url'`
  - 直接返回 URL
- 处理 `model.type === 'file'`
  - 使用 `LocalFileManager.loadModelFromFiles`
- 处理 `model.type === 'folder'`
  - 使用 `LocalFileManager.loadModelFromFolder`
- 管理 Object URL 生命周期
- 在模型源切换时自动清理旧资源

### 复用现有能力

该方案尽量复用已有实现，而非重写：

- `ThreeViewer`
- `CameraScriptController`
- `LocalFileManager`
- `ModelLoadResult`
- `CameraShot` / `CameraViewPreset`

## 事件与错误设计

### `onLoad`

模型加载成功后触发：

```ts
onLoad?: (result: ModelLoadResult) => void;
```

### `onLoadingChange`

模型准备和加载过程中触发：

```ts
onLoadingChange?: (loading: boolean) => void;
```

### `onError`

建议对错误来源做分层，便于第三方业务处理：

```ts
onError?: (
  error: Error,
  context: { stage: 'model-source' | 'model-load' | 'camera-script' }
) => void;
```

三个阶段说明：

- `model-source`
  - 本地文件或文件夹解析失败
  - 例如 GLTF JSON 无法解析
- `model-load`
  - Three.js 实际加载模型失败
- `camera-script`
  - `shot` 或 `preset` JSON 解析失败
  - 或脚本应用失败

## 尺寸与布局约定

组件不建议强制内置固定高度。

建议约定：

- 默认宽高继承容器
- 调用方自行提供 `style` 或 `className`

推荐示例：

```tsx
<ModelViewer
  model={{ type: 'url', url: '/model.glb' }}
  style={{ width: '100%', height: 480 }}
/>
```

这样更适合第三方项目接入，不会强行干扰宿主项目布局系统。

## SSR 与运行环境说明

该组件面向浏览器渲染场景。

需要说明以下边界：

- `File` 和 `File[]` 相关能力仅在浏览器中可用
- `Three.js` 渲染依赖 DOM 和 WebGL 环境
- 组件实现中应避免在模块顶层直接访问 `window`

如果第三方在 SSR 框架中使用，应在客户端侧渲染该组件。

## 导出设计

### 第一版建议

在 `src/index.ts` 中新增导出：

```ts
export { ModelViewer, type ModelViewerProps } from './components/ModelViewer';
```

由于当前包已经通过根入口导出，因此第一版理论上无需修改 `package.json` 的 `exports` 结构，也可以满足：

```tsx
import { ModelViewer } from 'threejs-viewer';
```

### 是否修改 `package.json`

第一版结论：

- 可以不新增子路径导出
- 只要把 `ModelViewer` 接入 `src/index.ts` 即可

后续如果希望支持更细粒度入口，再考虑增加：

- `threejs-viewer/model-viewer`

但这不属于当前必须项。

## Demo2 改造方案

`Demo2` 后续应从“实现页”调整为“示例页”。

### 改造前

`Demo2` 当前自己负责：

- 模型 URL / 文件 / 文件夹输入
- 本地资源文件处理
- `DemoViewer` 渲染
- `CameraScriptController` 接入
- 模型加载状态维护

### 改造后

`Demo2` 建议只负责：

- 展示各种输入控件
- 把输入整理为 `model`
- 把脚本整理为 `cameraScript`
- 将参数传给 `ModelViewer`

也就是说，展示区最终变成：

```tsx
<ModelViewer
  model={appliedModel}
  cameraScript={appliedCameraScript}
  grid={{ visible: true, size: 20, divisions: 20, plane: 'XZ', showAxes: true }}
  onLoad={...}
  onLoadingChange={...}
  onError={...}
/>
```

这样 `Demo2` 的角色会更清晰：

- 它是官方 demo 示例
- 不是底层实现承载体

## 第三方用户示例

### 1. URL 模型

```tsx
import { ModelViewer } from 'threejs-viewer';

export function UrlExample() {
  return (
    <ModelViewer
      model={{ type: 'url', url: 'https://example.com/model.glb' }}
      style={{ width: '100%', height: 480 }}
    />
  );
}
```

### 2. 本地文件模型

```tsx
import { useState } from 'react';
import { ModelViewer } from 'threejs-viewer';

export function FileExample() {
  const [file, setFile] = useState<File | null>(null);

  return (
    <>
      <input
        type="file"
        accept=".gltf,.glb"
        onChange={(e) => setFile(e.target.files?.[0] ?? null)}
      />

      <ModelViewer
        model={file ? { type: 'file', file } : null}
        style={{ width: '100%', height: 480 }}
      />
    </>
  );
}
```

### 3. 传入 preset 脚本

```tsx
import { ModelViewer } from 'threejs-viewer';

const preset = {
  version: 2,
  kind: 'orbit',
  target: { x: 0, y: 0, z: 0 },
  spherical: { radius: 8, phi: 1.1, theta: 0.8 },
};

export function PresetExample() {
  return (
    <ModelViewer
      model={{ type: 'url', url: 'https://example.com/model.glb' }}
      cameraScript={{ mode: 'preset', data: preset }}
      style={{ width: '100%', height: 480 }}
    />
  );
}
```

## 实施顺序

建议按以下顺序开发：

1. 新增 `ModelViewer` 组件
2. 新增 `useResolvedModel`，收口模型源解析逻辑
3. 在 `src/index.ts` 导出 `ModelViewer`
4. 为 `ModelViewer` 增加基础文档和示例
5. 将 `Demo2` 改造成 `ModelViewer` 的使用示例页

## 本期建议定稿项

本期建议确认以下三点：

1. 对外组件名使用 `ModelViewer`
2. 对外主 API 使用 `model + cameraScript`
3. 第一版通过包根入口导出，不强制新增 `package.json` 子路径导出

## 待确认问题

以下问题若你有明确偏好，可以在实现前一并定下来：

1. 是否需要对外暴露 `ref`
   - 例如继续暴露 `ThreeViewerHandle`
2. 是否需要内置空状态 UI
   - 还是完全交由宿主控制
3. 是否需要在第一版中提供更细粒度的错误码
   - 不只返回 `stage`

## 结论

本设计的核心思路是：

- 保留 `ThreeViewer` 作为底层能力
- 新增 `ModelViewer` 作为第三方 npm 用户的主入口
- 用 `model + cameraScript` 收敛复杂输入
- 让 `Demo2` 从实现页变成示例页

这套设计兼顾了三件事：

- 第三方接入成本低
- 对外 API 清晰稳定
- 最大限度复用现有实现，降低重构风险
